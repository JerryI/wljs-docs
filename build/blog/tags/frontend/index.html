Maintainer

  • on Graphics2D elements to listen users interactions
    • drag, click, zoom, mousemove
  • on Graphics3D elements
    • transform
  • on any EventObject
  • on the output cell using EvaluationCell[] to attach to cell's events

An event-driven approach was inspired by Javascript language, where one can subscribe to any changes of any objects. Here it is way more simplified compared to JS, but covers all practical cases.

Introducing NotebookStore

August 17, 2023 · One min read
Kirill Vasin
Maintainer

Sometimes it is easier to ship notebook together with the data it depends on. However, not as a text encoded using BASE64 pasted into a notebook cell - it would just kill the performance, but as a dedicated inner structure, that can be exposed using NotebookStore

NotebookStore["key"] = 1+1
NotebookStore["key"] === 1+1

As simple as that. This is not real association, but mimics its behavior. To get all keys, use 

Keys[NotebookStore]

In the worst case scenario one can use Iconize, but that will make the data to be stored inside a notebook cell, which is recommended.

tip

For the arrays of data apply Compress / Uncompress to save up the space

Introducing Events shortcuts

June 10, 2023 · One min read
Kirill Vasin
Maintainer

One of the paradigm of WLJS Frontend is an event-based approach. Any sliders, buttons are event generators, that produce EventObject[<|...|>]. In order to simplify or mimic Mathematica's way of binding symbols to input elements Assign keyword was introduced.

Instead of 

InputRange[0,1, 0.1]
EventHandler[%, Function[data, x = data]]

one can write

InputRange[0,1, 0.1] // Assign[x]

The result will be the same
